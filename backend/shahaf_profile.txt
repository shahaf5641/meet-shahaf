## ניסיון תעסוקתי

### התמחות הנדסת תוכנה — Hexagon ALI, נשר (9 חודשים, 2025)

**על החברה והמוצר:**
עבדתי על מוצר פנימי בשם HCAP — פלטפורמה מרכזית לניהול ארגונים, משתמשים והרשאות עבור כלל מוצרי Hexagon. המערכת מממשת RBAC (Role-Based Access Control), פועלת כשכבת תשתית במערכת Multi-Tenant, ומשמשת כ-Single Source of Truth לניהול זהויות וגישה (IAM). המוצר כלל אינטגרציה עם SSO ועבד כ-Event-Driven Architecture.

**סביבת העבודה:**
- צוות של ~10 איש, עם שני צוותים נוספים — אחד בנשר ואחד בהודו
- מתודולוגיית Agile עם ספרינטים חודשיים
- עבדתי בצמוד ל-SDET שלי — הייתי יד ימינו ביומיום
- שפות וכלים: .NET / C# (עיקרי), Microservices, REST APIs, Postman, Docker, Azure, YAML

**מה עשיתי — שלושה תחומים מרכזיים:**

**1. Refactor ותחזוקת טסטים קיימים**
ביצעתי refactor לטסטים קיימים שהיו שבורים בגלל עדכוני מוצר, עדכנתי אותם כדי שיתאימו למצב הנוכחי של המערכת, ושיפרתי את תחזוקתיות הקוד ועקביותו.

**2. Hooks Health Check — ייעול pipeline ב-Azure DevOps**
המערכת הייתה מבוססת Event-Driven Architecture, ולכן רוב הטסטים היו תלויים ב-Hooks. כאשר ה-Hooks לא עבדו, ה-Pipelines בשלמותם היו נופלים לאחר ריצה של 20–30 דקות — כל הטסטים נכשלו לא בגלל באגים במוצר, אלא כי ה-Hooks לא היו זמינים. הדבר פגע קשות בסטטיסטיקות Azure Tests.
יזמתי מפגש עם ה-SCRUM Master וה-SDET שלי, ויחד הגדרנו פתרון: יצרתי טסט ייעודי — **Hook Health Check** — שבודק את זמינות ה-Hooks באופן מבודד, ללא תלויות אחרות. הוספתי תנאי ב-YAML של ה-Pipelines: הפייפליין מתחיל לרוץ **רק אם** ה-Hook Health Check עבר בהצלחה; אחרת — ה-Pipeline מבצע Skip ונחסך הזמן. כך גם הסטטיסטיקה הוגנת, וגם מזהים מיידית כשיש בעיית Hooks.

**3. Role Tests — תשתית בדיקות הרשאות מאפס (המשימה הכי גדולה)**
המערכת כללה Unit/Integration/Synthetic/Load/Performance tests — אבל לא הייתה שום בדיקה על הרשאות משתמשים. קיבלתי משימה לפתח תשתית בדיקות RBAC מאפס, לבד.
קיבלתי טבלת Truth Table (Excel) שמרכזת את כל ההרשאות של כל תפקיד לכל Action על כל אובייקט במערכת.
מה שעשיתי:
- יצרתי אובייקטים במערכת וחיברתי ביניהם דרך Relationships כדי לבדוק פעולות שמחייבות קישור בין אובייקטים
- לכל אובייקט בניתי טסטים שבודקים את כל הפעולות עליו עבור כל סוג משתמש
- השוויתי בין תגובות ה-API (JSON responses) לבין הצפוי מה-Truth Table, והצגתי לוגים ברורים לכל MISMATCH
- עבדתי עם Tokens, מנגנוני Security, ו-OData Queries מול ה-SDX Database (מוצר פנימי של החברה)
המשימה הפחידה אותי בהתחלה — הייתה גדולה ולא פשוטה. אבל נכנסתי אליה, נתקלתי בקשיים, פתרתי חלק לבד וחלק עם הצוות, וסיימתי אותה בהצלחה.

### AI Recruiter Avatar (2025 — פרויקט נוכחי)

**מה זה:**
פרויקט אישי שבניתי לבד — סוכן AI קולי שמייצג אותי מול מגייסים. במקום שמגייס יצטרך לתאם איתי ראיון, הוא נכנס לקישור, רואה אווטר תלת-מימד שלי, ומדבר איתו בשיחה קולית בזמן אמת — 24/7, בלי תיאום.

**איך זה עובד:**
המגייס נכנס לקישור, מכניס את דרישות המשרה (טקסט או PDF), ומתחיל שיחה קולית עם האווטר. הסוכן עונה בעברית, בגוף ראשון, כאילו הוא אני — על בסיס הפרופיל האמיתי שלי ועל בסיס דרישות המשרה שהמגייס הכניס.

**ארכיטקטורה:**
- Frontend: React + Three.js (אווטר GLB תלת-מימד), AudioWorklet לצילום PCM16 בזמן אמת
- Backend: Python FastAPI, WebSocket proxy
- AI: OpenAI Realtime API (gpt-4o-realtime-preview) — audio-in, audio-out, ללא STT/TTS נפרד
- Audio pipeline: Float32 → Int16 PCM16 → base64 → OpenAI → audio chunks → AudioContext scheduling
- אווטר: head tracking לפי עכבר המגייס, mouth morphs לפי amplitude של האודיו

**למה בניתי את זה:**
רציתי להוכיח יכולת בצומת שבין AI, real-time audio ו-3D web. בניתי את זה מאפס לבד, כולל פתרון בעיות טכניות לא טריוויאליות — WebSocket proxy, PCM16 encoding, audio scheduling ללא glitches, ואנימציית אווטר תגובתית.
